# Architecture Decision Records

## 철학
MVP 속도와 운영 부담 최소화가 최우선. 사용자 수가 늘어도 내가 비용을 부담하지 않는 구조를 택한다. 서버 없음, DB 없음, 로그인 없음 — 모든 책임은 사용자 브라우저에. 의심스러우면 단순한 쪽으로.

각 ADR은 **결정 / 이유 / 트레이드오프 / 대안 / 재검토 조건** 형식.

---

### ADR-001: BYOK (사용자가 본인 API key 입력) 채택
**결정**: YouTube Data API key와 Anthropic API key를 사용자가 직접 발급받아 입력하고, LocalStorage에 저장한다.
**이유**:
- 서버 불필요 → 정적 배포만으로 운영 가능.
- 트래픽이 늘어도 내가 API 비용을 부담하지 않는다.
- 사용자가 본인 사용량을 직접 통제·확인.
**트레이드오프**: 온보딩 무거움 (API key 2개 발급). 기술 친화적 사용자만 사용 가능. → 발급 가이드 링크 명확히 제공해 완화.
**대안 검토**:
- Vercel/Cloudflare serverless function으로 키 보호 → 사용자 경험은 좋지만 내 비용 부담. 운영 시작 시점에 다시 검토.
- 백엔드 풀스택 → MVP 단계에서 과함.
**재검토 조건**: 일반 사용자 1,000명 이상 도달 시, 또는 "key 발급이 어려워서 못 쓴다"는 피드백이 30% 이상 누적될 때.

### ADR-002: Claude (Anthropic) 단일 모델 — `claude-haiku-4-5`
**결정**: 청크 분석과 종합 단계 모두 `claude-haiku-4-5` 단일 모델로 처리.
**이유**:
- **비용 최우선**. BYOK 구조라 비용은 사용자가 부담 → 가벼울수록 좋다.
- Haiku 4.5는 한국어·영어 모두 1차 품질 보장 (서비스 타깃 언어).
- 단일 모델이면 비용 추정·캐싱·rate limit 처리 모두 단순화.
- Anthropic SDK가 브라우저 직접 호출 공식 지원 (`dangerouslyAllowBrowser`).
**트레이드오프**: 종합 단계의 자연스러운 한국어 표현이 Sonnet 4.6 대비 다소 떨어질 수 있음. → 프롬프트 엔지니어링으로 보완 (예시·톤 가이드 제공). 추론이 필요한 복잡한 비교/추리는 약함 → 종합 프롬프트를 "추론보다 정리·요약" 위주로 구성.
**대안 검토**:
- 2단계 (Haiku + Sonnet) → 종합 품질 ↑이지만 비용 약 3~5배. 사용자 부담.
- GPT-4o-mini → 한국어 품질 Haiku 4.5와 동등 또는 약간 낮음. SDK 호환 별개.
- Gemini Flash → 무료 tier 매력적이나 BYOK가 OAuth라 복잡.
**재검토 조건**:
- 종합 결과 품질이 떨어진다는 피드백 3건 이상 누적 시 → 종합만 Sonnet으로 업그레이드 옵션 제공.
- Anthropic이 더 저렴한 모델 출시 시.

### ADR-003: Next.js 15 App Router + 정적 export
**결정**: Next.js 15 App Router + Tailwind 4. 모든 페이지 `"use client"` 기본. 서버 컴포넌트는 레이아웃 셸만. `next.config.ts`에서 `output: "export"`로 정적 사이트 빌드.
**이유**:
- 향후 serverless function 도입(공유 링크, 결제 등) 시 매끄러운 확장 가능.
- 파일 기반 라우팅, 코드 스플리팅, Image 최적화 기본기.
- Tailwind 4와 App Router 조합이 가장 안정적이고 사례 많음.
**트레이드오프**: 순수 Vite 대비 빌드 무거움. SSR/RSC 강점 절반은 사용 안 함.
**대안 검토**:
- Vite + React: 더 가볍지만 라우팅·메타데이터·이미지 처리를 별도 구성해야 함.
- Astro: 정적 사이트에 최적이지만 인터랙티브 비중이 80% 이상이라 이점 사라짐.
- 순수 HTML/JS: 너무 원시적, 컴포넌트화 어려움.
**재검토 조건**: Tailwind 4 또는 React 19 호환성 이슈가 Next.js에서 1주 이상 미해결될 때.

### ADR-004: LocalStorage를 유일한 저장소로 사용
**결정**: API key, 리포트, 설정 모두 LocalStorage. IndexedDB·Cookies·서버 저장 없음.
**이유**: 리포트 한 건 평균 5~20KB → LocalStorage 5MB 안에서 수백 건 보관. API 단순(`getItem/setItem`).
**트레이드오프**: 브라우저·기기 간 동기화 불가. 캐시 지우면 사라짐. 5MB 한도 존재.
**대안 검토**:
- IndexedDB → 한도 더 넉넉하지만 비동기 API라 코드 복잡, MVP 규모에 과함.
- Origin Private File System → 브라우저 지원 편차 큼.
**재검토 조건**: 사용자가 100건 이상 리포트를 보관하기 시작하거나, "기기 간 동기화" 요청이 누적될 때.

### ADR-005: 답글(reply) 분석 제외, top-level 댓글만 사용
**결정**: `commentThreads.list`의 `topLevelComment`만 수집. `replies`는 무시.
**이유**: 답글은 대부분 작성자 본인 또는 댓글러끼리의 대화 → 피드백 신호 약함. API 쿼터 절약.
**트레이드오프**: 답글의 보석 같은 피드백 놓침.
**대안 검토**:
- 사용자가 옵션으로 켤 수 있게 → Phase 2 후보.
**재검토 조건**: "답글까지 보고 싶다"는 피드백 3건 이상.

### ADR-006: 댓글 수집 상한 500개, 정렬 relevance
**결정**: 한 영상당 최대 500개 (5페이지 × 100개). YouTube API `order=relevance` (기본값).
**이유**:
- 500개면 sentiment 분포와 토픽이 통계적으로 안정.
- 쿼터 비용 5 unit으로 가벼움.
- Claude 토큰 합계 30~45K 수준 → Haiku 4.5 한도 내.
- `relevance`는 YouTube가 좋아요 수와 답글 활성도를 기준으로 정렬해 의미 있는 댓글 우선.
**트레이드오프**: 댓글 수천 개 영상에서 long-tail 의견 누락. 최신 댓글이 누락될 수 있음.
**대안 검토**:
- `order=time` (최신순): 시의성 있는 피드백에 유리하나 스팸 비중 높음.
- 1,000개 이상 수집: 비용 + 시간 부담 증가, 품질 향상 마진 작음.
**재검토 조건**: 평균 분석 영상의 댓글 수가 800개 이상으로 관측될 때, 또는 정확도 부족 피드백 누적.

### ADR-007: 패키지 매니저는 pnpm
**결정**: pnpm 사용. `packageManager` 필드 고정.
**이유**: 디스크 효율(symlink 기반), 설치 속도, monorepo 대응. CI 캐시 효과 큼.
**트레이드오프**: 일부 패키지 호환성 이슈 (`peerDependencies` 엄격).
**대안 검토**: npm(느림, 디스크 비효율), yarn berry(설정 복잡).
**재검토 조건**: 호환성 이슈로 1주 이상 막힐 때.

### ADR-008: 테스트는 Vitest + Testing Library + MSW
**결정**: 단위/통합은 Vitest, 컴포넌트는 React Testing Library, API 모킹은 MSW. E2E는 MVP에서 제외.
**이유**:
- Vitest는 Vite 기반이라 빠르고 TS 친화적. Jest 호환 API.
- MSW로 YouTube/Anthropic 모킹 → 실 API 호출 없이 파이프라인 검증.
- E2E 없이도 핵심 파이프라인은 통합 테스트로 충분.
**트레이드오프**: 실 환경 회귀 못 잡음 → 수동 QA 보완 필요.
**대안 검토**: Jest (느림), Playwright(좋지만 MVP 부담).
**재검토 조건**: 회귀 버그 3건 이상 누적 시 E2E 도입.

### ADR-009: TypeScript strict mode 강제
**결정**: `tsconfig.json`에서 `"strict": true` + `"noUncheckedIndexedAccess": true`.
**이유**: 도메인 모델이 핵심이라 타입 안정성 중요. LLM 응답을 다루므로 `undefined` 누락 케이스가 흔함.
**트레이드오프**: 초기 작성 시간 증가.
**재검토 조건**: 없음 (이 결정은 되돌리지 않는다).

### ADR-010: 분석 결과 캐싱 정책
**결정**:
- 같은 `videoId`로 24시간 이내 분석한 리포트가 있으면 "기존 리포트를 보시겠습니까? 또는 재분석하시겠습니까?" 선택 화면 표시.
- 자동 재사용은 하지 않음 — 사용자 의도 우선.
**이유**: 댓글은 시간에 따라 늘어나므로 무조건 캐시 재사용은 부정확. 그러나 같은 영상을 1분 만에 두 번 분석하는 실수는 막아야 함.
**트레이드오프**: UX에 한 단계 추가.
**대안 검토**: 무조건 새로 분석(비용 낭비), 강제 캐시(부정확).
**재검토 조건**: 사용자가 "재분석 옵션이 헷갈린다" 피드백 시.

### ADR-011: temperature 0.2 (청크 분석) / 0.3 (종합)
**결정**:
- 청크별 분석: `temperature: 0.2` (분류 작업, 결정성 우선)
- 종합 단계: `temperature: 0.3` (자연스러운 한국어 문장 우선)
**이유**: 완벽한 결정성은 어차피 LLM에서 보장 불가지만, 낮은 temperature가 분포 안정에 유리.
**트레이드오프**: 다양성 감소 (같은 표현이 반복될 수 있음).
**대안 검토**: 0 (결정성 최대, 그러나 너무 robotic), 0.7 (창의적이지만 분석에는 노이즈).
**재검토 조건**: 동일 입력에 sentiment 분포가 10% 이상 흔들리는 사례 발견 시.

### ADR-012: 프롬프트 캐싱 사용 (Anthropic prompt caching)
**결정**: 청크 분석의 system 프롬프트는 `cache_control: { type: "ephemeral" }`로 마킹.
**이유**: 5번 청크 호출 시 동일한 system 프롬프트가 5번 전송됨 → 캐싱 시 비용 90% 절감, 첫 호출 후 latency 단축.
**트레이드오프**: 캐시는 5분 TTL이라 영상 분석이 5분 이상 걸리면 만료. 보통은 3분 내라 영향 없음.
**대안 검토**: 캐싱 안 함 (단순하지만 비용 손해).
**재검토 조건**: Anthropic이 캐싱 API를 deprecate하거나 가격 모델을 변경할 때.

### ADR-013: 청크 크기 100개, 동시성 최대 3
**결정**: 댓글 100개씩 청크. `Promise.allSettled` + concurrency limit 3.
**이유**:
- 100개는 Haiku 4.5 컨텍스트와 응답 품질의 균형점.
- 동시성 3은 사용자 tier의 RPM/TPM 한도 일반 케이스에 안전.
- `allSettled`로 일부 청크 실패 시 나머지로 진행 가능.
**트레이드오프**: 한 청크 실패 시 해당 100개 댓글이 분석에서 빠짐. → fallback으로 누락 청크 1회 재시도.
**대안 검토**:
- 청크 크기 50: 호출 수 증가로 latency 증가.
- 청크 크기 200: 응답 품질 떨어짐 (모델이 일부 댓글 누락).
- 직렬 호출: 너무 느림.
**재검토 조건**: tier 낮은 사용자에서 일관되게 429 발생 시 동시성을 2 또는 1로 조정.

### ADR-014: 댓글 정규화는 보존형
**결정**: 원문(`text`)과 정규화 본문(`textNormalized`)을 모두 저장. 분석은 정규화 본문 사용. 인용은 원문 사용.
**이유**:
- 정규화: URL/멘션 제거로 LLM이 광고 댓글 영향 덜 받음.
- 원문 보존: 인용을 정확히 보여줄 수 있음. LLM이 댓글을 새로 지어내는 hallucination 방지.
**트레이드오프**: 저장 용량 약간 증가.
**대안 검토**: 정규화만 저장(인용 불가능), 원문만 저장(LLM 노이즈).
**재검토 조건**: 없음.

### ADR-015: 스팸은 필터링하되 통계는 보존
**결정**: 휴리스틱 스팸 필터로 분석에서 제외하지만, `stats.spamFiltered`로 개수 보존.
**이유**: 스팸을 포함하면 sentiment 분포가 왜곡 ("구독해주세요"는 neutral 잘못 분류). 그러나 사용자에게는 "내 댓글 중 스팸이 얼마나 되는가"도 정보 가치.
**트레이드오프**: 휴리스틱이 정상 댓글을 false positive로 제외할 수 있음.
**대안 검토**:
- 필터링 안 함 → 분석 노이즈.
- LLM으로 스팸 판정 → 비용 + latency 증가.
**재검토 조건**: false positive 사례가 사용자 피드백으로 들어올 때.

### ADR-016: 인용 hallucination 검증 필수
**결정**: LLM이 반환한 인용은 반드시 `commentId` 참조 형태로 받고, 실제 수집 댓글에 존재 여부 검증. 매칭 실패 시 인용 제외.
**이유**: 가장 위험한 hallucination = "있지도 않은 댓글을 인용". 사용자 신뢰 결정타.
**트레이드오프**: 모델이 약간 다른 문구로 paraphrase하면 매칭 실패 → 인용 후보 줄어듦.
**대안 검토**: 인용 원문을 그대로 받기 → 검증 어렵고 hallucination 막기 힘듦.
**재검토 조건**: 매칭 실패율이 20% 이상이면 프롬프트 강화 또는 매칭 알고리즘 개선.

### ADR-017: 언어 정책 — 인터페이스 한/영, 댓글 분석 한/영 1차 보장
**결정**:
- **인터페이스 언어**: 한국어와 영어 모두 지원. 기본은 한국어, 브라우저 `navigator.language` 자동 감지 + 상단 토글로 수동 전환. i18n 키 기반 (`messages/ko.json`, `messages/en.json`).
- **댓글 분석 언어**: 한국어·영어 1차 품질 보장. 그 외 언어는 best-effort. 댓글 언어 감지는 하지 않고 LLM에게 맡김 (Haiku 4.5가 두 언어 모두 잘 처리).
- **리포트 출력 언어**: 인터페이스 언어를 따라감 (UI가 한국어면 한국어, 영어면 영어). 댓글 인용은 원문 유지.
**이유**:
- 한국 크리에이터 + 영어권 크리에이터 모두 잠재 사용자.
- 채널 댓글이 한·영 혼재인 경우(글로벌 콘텐츠)에도 분석이 자연스러움.
- i18n 인프라(파일 분리)는 초기에 깔아두면 추가 언어 확장 비용 낮음.
**트레이드오프**: i18n 도입으로 컴포넌트마다 `t(key)` 호출 추가. 일본어·중국어 등은 분석 품질이 떨어질 수 있음.
**대안 검토**:
- 인터페이스 한국어 only → 영어권 사용자 진입 차단. 1차 목표(개인 사용 + 한국 크리에이터)에는 충분하지만 확장성 막힘.
- 사전 언어 감지 후 분기 → Phase 2.
**재검토 조건**:
- 일본어·중국어 등 비한영 사용자 비중 20% 이상 → 해당 언어 추가.
- i18n 라이브러리 (예: `next-intl`) 호환성 이슈 발생 시.

---

### ADR-018: 콘텐츠 안전 — 혐오 표현은 인용 제외, 욕설은 마스킹
**결정**:
- 인용 대상에서 명백한 혐오 표현(차별, 폭력 위협)은 LLM 프롬프트로 제외 지시.
- 일반 욕설은 인용 가능하되 표시 시 `*` 마스킹.
**이유**: 크리에이터에게 도움 되는 피드백 추출이 목적. 화면에 자극적 표현 그대로 띄울 필요 없음.
**트레이드오프**: 혐오 표현이 빈번한 채널의 경우 부정 의견 분포가 과소평가될 위험. → 통계에는 포함하고 인용에만 제외.
**대안 검토**: 모두 그대로 표시 → UX 위험. 모두 제외 → 분석 왜곡.
**재검토 조건**: 인용 부족으로 리포트 품질 떨어지는 사례 누적 시.

### ADR-019: 진행 상태는 클라이언트 측 reducer (스트리밍 미사용)
**결정**: 진행 단계는 `progressReducer`로 관리. Anthropic 스트리밍 API는 MVP에서 사용하지 않음.
**이유**:
- 청크 분석은 응답 전체를 받아야 검증 가능 → 스트리밍 이점 적음.
- 사용자에게는 "청크 N/M 완료" 단위로 진행 표시 → 충분히 부드러움.
- 스트리밍 도입 시 에러 처리·취소 복잡.
**트레이드오프**: 각 청크 응답이 1회 chunk로 도착 → "글자가 하나씩 떠오르는" UX는 없음.
**대안 검토**: 스트리밍 사용 → 향후 Phase 2.
**재검토 조건**: 사용자가 진행감 부족하다고 피드백 시.

### ADR-020: 비용 표시 정책
**결정**:
- 분석 시작 전: 댓글 수 기반 **예상** 비용 표시 (`$0.03~$0.05`).
- 분석 완료 후: 실제 사용 토큰 기반 **실측** 비용 표시.
- 모두 USD 표시. 환율 변환 안 함.
**이유**: 비용 투명성. BYOK 구조에서 사용자가 자기 비용을 알 권리.
**트레이드오프**: 단가 변동 시 코드 수정 필요 (`lib/claude/costEstimate.ts`).
**대안 검토**: 비용 표시 안 함 → BYOK에서는 무책임.
**재검토 조건**: Anthropic 가격 모델 변경 시.

### ADR-021: 에러 추적은 외부 도구 없이 로컬 콘솔
**결정**: Sentry 등 외부 에러 추적 도구 사용 안 함. 사용자 콘솔에 로그 출력 + GitHub Issues 통한 수동 리포트.
**이유**:
- 외부 도구 사용 시 사용자 데이터(댓글 본문)가 외부로 전송될 위험.
- BYOK 철학 = 사용자 데이터는 사용자 브라우저에만.
**트레이드오프**: 오류 발생을 자동으로 알 수 없음.
**대안 검토**: Sentry 자체 호스팅, 사용자 옵트인 에러 전송 → Phase 2.
**재검토 조건**: 사용자가 1,000명 이상이고 디버깅 어려움 호소 시.

### ADR-022: LocalStorage 한도 정책 (LRU 권유, 자동 삭제 안 함)
**결정**:
- 사용량 80% 도달 시: 사용자에게 알림 + 오래된 리포트 5개 후보 제시 (사용자 확인 후 삭제).
- 100% 도달 시: 새 분석 차단 + 모달로 정리 유도.
- **자동 삭제는 절대 안 함** — 사용자 동의 필수.
**이유**: 리포트는 사용자가 시간 들여 만든 자산. 동의 없이 지울 수 없음.
**트레이드오프**: 한도 도달 시 분석 못 함 → 사용자 짜증.
**대안 검토**: 자동 LRU eviction → 데이터 손실 위험.
**재검토 조건**: 사용자가 "자동 삭제 옵션을 켤 수 있게 해달라"고 요청 시 옵션 추가.

### ADR-023: 사용자 텔레메트리 / 분석 도구 미사용
**결정**: Google Analytics, Plausible 등 어떤 분석 도구도 사용하지 않음.
**이유**:
- BYOK 철학과 모순.
- 정적 사이트이고 사용자 신뢰 = 핵심.
- 사용자 행동은 인터뷰로 직접 청취.
**트레이드오프**: 사용자 행동 데이터 없음 → 제품 개선 의사결정이 추측 기반.
**대안 검토**: 자체 호스팅 분석(서버 필요해서 모순).
**재검토 조건**: 사용자가 1,000명 이상이고 정량 의사결정 필요할 때 — 옵트인 형태 검토.
